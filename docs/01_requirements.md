# 01 機能要件(mfg)

## 目的

中小製造業(オーダーメード・受注生産)の、Webからの見積依頼受付と
受注管理、Office(OnlyOffice)への台帳連携。WordPress サイトの
置き換え先。決済は持たない(商流は従来のまま)。

## ユーザー種別

| 種別 | 説明 | 使う画面 |
|---|---|---|
| 訪問者 | 検索から来る見込み客 | 公開静的サイト(カタログ) |
| 顧客(customer) | 取引先の担当者。登録して見積依頼 | 顧客アプリ |
| 社内スタッフ(staff) | 見積回答・受注管理。担当名(contact_label) | 社内アプリ(Flet・DB直結) |
| admin | 製品・ユーザー管理 | 社内アプリ(管理メニュー) |

## 機能一覧

### 公開静的サイト(Python生成・Cloudflare Pages)
- P-01 会社案内・製品カタログ・事例。products の公開分から静的HTML生成
- P-02 各製品ページに「見積を依頼する」→ 顧客アプリへ遷移
- P-03 SEO はここが担う(Flutter は使わない)

### 顧客アプリ(Flutter: Web / iOS / Android)
- C-01 登録・ログイン(PocketBase。メール確認)
- C-02 カタログ閲覧(静的JSONから。アプリ機停止中も閲覧可)
- C-03 見積カート: 製品+数量+個別仕様(色・寸法等)を追加
- C-04 見積依頼送信(quotes 作成・見積番号採番・会社へメール通知)
- C-05 やり取り: メッセージ(添付可: 図面・仕様書PDF)。回答の受領
- C-06 発注意向の表明(status→ordered。正式な契約・支払いは従来の商流)
- C-07 マイページ: 依頼一覧(状態: 依頼中/回答済み/受注/完了)
- C-08 QR読み取り(/scan): 銘板・カタログ紙面のQRから製品ページへ直行

### 社内アプリ(Flet / Python。DB直結、会社サーバー内で稼働)
- S-01 見積対応: requested を上に。品目・数量・要望を見て回答
      (answered_at / answered_by〔担当者〕を記録)
- S-02 受注・完了の管理(ordered_at 記録=台帳の計上基準)
- S-03 製品管理: 追加・編集・公開/非公開。保存時に静的カタログ再生成
- S-04 在庫・価格の参照: **基幹API から取得して表示のみ**(本システムは
      持たない。基幹が無い会社は「要見積」表示で運用開始できる)
- S-05 xlsx エクスポート: 見積台帳・受注台帳(見積番号・受付/回答/受注日・
      担当・相手・品目・数量)。OnlyOffice でそのまま開ける
- S-06 QRラベル: 製品QR(銘板・現物・紙カタログ用)の一括PDF生成
- S-07 ユーザー管理(admin): 顧客の承認・凍結、スタッフの担当名

## QR の用途(会社にも顧客にも便利)

| 貼る場所 | 飛び先 | 効果 |
|---|---|---|
| 機械の銘板・製品現物 | 製品ページ | 型番を調べずに仕様・図面へ。**保守・修理依頼の入口**(そのまま見積依頼できる) |
| 紙カタログ・展示会パネル | 製品ページ | 紙からアプリへの導線 |
| 見積書・納品書 | 見積番号ページ | 案件の現状確認・問い合わせ照合 |

## 非機能・原則

- 在庫・価格の正は会社側基幹(API参照のみ)。内数で足り、テスト段階は
  正確な数を表示
- 見積番号は DB全体の通し番号(年+連番)
- メールは専用機で常時稼働、アプリ機は計画停止可
- カタログは静的配信でアプリ機停止中も閲覧可、API不応答時は
  Widget の error 状態で「メンテナンス中」
- 責任分界: サーバー・データ・メール・OnlyOffice の所有と運用は会社。
  当方は構築・引き継ぎ研修(時給1万円)と任意の運用支援(月1万円)
- 商材ごとの法規制(建材の性能表示等)は案件ごとに確認する(先回りで
  作り込まない)
