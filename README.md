# 製造業向け 見積・受注アプリ プロジェクト一式(mfg)

中小製造業(オーダーメード・受注生産)向けの、製品カタログ+見積依頼+
受注管理+Office 連携のシステム。WordPress サイトの置き換え先としても
提案する。決済は持たない(商流は従来のまま)。

## 3つの画面と役割分担

| 画面 | 技術 | 役割 |
|---|---|---|
| 公開静的サイト | Python 生成の静的 HTML(Cloudflare Pages) | 会社案内・製品カタログ。SEO はここが担う |
| 顧客アプリ | Flutter(iOS / Android / Web) | 見積カート→見積依頼→やり取り→発注意向。QR 読み取り |
| 社内アプリ | Flet / Python(DB 直結・社内のみ) | 見積対応・製品管理・台帳 xlsx・QR ラベル |

## 設計原則

- **在庫・価格の正は会社側基幹**: 本システムは持たず、会社の基幹 API
  (標準 HTTP+JSON)から取得して表示のみ。基幹が無い会社は「要見積」
  表示で運用開始し、基幹 API ができたらつなぐ(段階導入)
- **見積番号は DB 全体の通し番号**(年+連番)。基幹・会計の伝票と照合できる
- **帳票はファイル直接生成**: xlsx を openpyxl で直接書き出し、OnlyOffice で
  開く。Office 連携 API・帳票製品・追加ライセンスは使わない。
  帳票の追加は「SELECT+整形の関数を1本足す」だけ
- **QR で紙・実物とつなぐ**: 機械の銘板→製品ページ(そのまま保守・修理の
  見積依頼へ)、紙カタログ→製品ページ、見積書→案件照合
- **決済なし**: 見積・受注・支払いは当事者間(従来の商流)
- **仕様変更は安い前提で作る**: コードは AI が書く時代であり、仕様書を
  直して実装を追随させるコストは低い。会社ごとの運用差を見越した
  設定項目・抽象層・拡張ポイントは作らず、いま決まっている運用に合う
  最も単純な形で実装する。1社1式なので、導入先ごとに仕様書ごと
  変えてよい(人がコードを書いていた時代の「変更に備える設計」は不要)

## 導入モデル(1社1式・自営)

- 会社ごとに自営サーバーへ導入(シングルテナント)。会社がサーバー・
  データ・メール・OnlyOffice を所有・運用し、当方は構築・引き継ぎ研修
  (時給1万円)と任意の運用支援(月1万円)を提供する。
  取引ごとの手数料は取らない
- 責任分界: サーバー・データ・基幹・在庫の正は会社。当方は仕組みの提供と
  契約範囲の支援(24時間対応は約束しない)

## 将来拡張(最小基幹への道)

受注台帳の先に、商品・在庫・受発注・帳簿の核(仕訳・元帳・試算表)まで
自作できる(受注確定→売掛/売上の仕訳自動生成)。**税務計算・申告・給与
など法改正で動く部分は作らず**、仕訳 CSV で会計 SaaS / 税理士へ渡す。
自作帳簿を税務の正本にするかは税理士と決める(管理会計用途が基本)。
